# Facts/Template Migration Plan: `knicks_2026_historic` + `player_ranking_overview`

Plan only. Nothing here has been executed. The goal is to bring both projects onto the same single-source-of-truth facts/template system that `home_court/` already uses, so every number cited in prose is rendered from `*_facts.json` (generated by the analysis pipeline) instead of being hand-typed, and guard tests fail when prose and data drift apart.

**Execution model: this plan is built to be run by Sonnet sub-agents.** Every work unit below is tagged with a `→ subagent` block naming the agent, its model (`sonnet`), and whether it runs serially or in parallel with siblings. Phase 0 and Phase 1 are serial (they edit shared files). Phase 2 is fully parallel (one sub-agent per doc, no shared file edits). Spawn agents only at the points marked; the orchestrator (you) synthesizes their results and runs the gates between phases.

**Git command convention: never `cd` for git.** Run every git command with `git -C <repo-or-worktree-path> <subcommand>` (e.g. `git -C "$WT" status`, `git -C "$WT" add -A`, `git -C "$WT" commit -m ...`), not `cd <path> && git ...`. A `cd` inside a compound command triggers a permission prompt on each call; `git -C` does not, so this keeps the worktree merges and incremental commits below from prompting for approval repeatedly. The same applies to running the pipeline/tests from a worktree: pass the path explicitly rather than `cd`-ing in.

## 0. Reference: how the `home_court` system works

This is the target architecture, distilled from `home_court/FACTS_MIGRATION.md`, `home_court_facts.py`, `render_docs.py`, and the guard tests.

### The `Fact` model
`<project>_facts.py` defines:

```python
@dataclass
class Fact:
    value: object       # numeric (int/float) OR a pre-written plain-language string
    fmt: str | None     # Python format spec, e.g. "{:+.3f}" or "{:.0%}"
    unit: str | None    # appended after the number, e.g. "pp/yr"
    note: str | None    # description for the reference table
    @property
    def display(self) -> str: ...   # renders value+fmt+unit, or passes a string through
```

A `Facts` singleton (`FACTS = Facts()`) holds a dict of named `Fact`s plus a `_guards` dict for qualitative claims. Two fact kinds:
- **Numeric** — `value` is a number; `fmt`+`unit` control display (`-0.2438, "{:+.3f}", "pp/yr"` → `"-0.244 pp/yr"`).
- **Plain-language** — `value` is a string authored next to the computation (e.g. `"a quarter of a point"`), so editorial roundings live in code and can be re-judged when the number moves.

### Registration and generation
Inside each `run_*` analysis function, next to the existing `print()` that writes to `results.md`, add a sibling `FACTS.set(name, value, fmt, unit=..., note=...)`. Qualitative claims get `FACTS.guard(name, condition, claim, value)`. At the end of `run()`:

```python
FACTS.dump(_FACTS_PATH)          # docs/<project>_facts.json   (committed)
FACTS.dump_guards(_GUARDS_PATH)  # docs/<project>_guards.json  (committed)
```

Naming convention: dot-separated namespaces (`reg.slope`, `share.fouls.decline`). `home_court` ended at 266 facts / 7 guards.

### The template/render mechanism
Docs become Jinja templates `docs/<doc>.md.j2`. Placeholder syntax (non-standard delimiters so they don't collide with Quarto's `{{`/`{%`/`{#`):

```
<< f("fact.name") >>                  # render the fact's display string
<< f("ref.x", "{:.2f}") >>            # precision override on the raw value
<< f("reg.slope") |tm >>              # ASCII hyphen → typographic minus (U+2212)
```

`render_docs.py`:
- `_make_env(facts_path, annotate)` — loads the JSON, defines `f(name, fmt=None)` (raises `KeyError` on unknown names), builds a Jinja2 `Environment` with `<< >>` / `<% %>` / `<# #>` delimiters and `StrictUndefined`.
- `render_all()` — globs `docs/*.md.j2`, renders each to `docs/<doc>.md`.
- `--watch` (re-render on mtime change), `--reference` (write `docs/<project>_facts_reference.md` lookup table), `--annotate` (render to `*.annotated.md` with `value [name]` so a reviewer can confirm the right fact is wired to each number).

`render_all()` is called at the top of `generate_report.py`'s `__main__`, so `python3 generate_report.py` re-renders templates before building the PDF.

### The guard tests
- `tests/test_facts_match_results.py` — for each numeric fact, searches `results.md` for the value at several precisions/signs; fails if a fact's value is absent from `results.md` (catches `FACTS.set()` drifting from its sibling `print()`). String facts and an `ALLOW_ABSENT` set are skipped.
- `tests/test_prose_claims.py` — loads `*_guards.json`, asserts every guard is `"ok": true`.
- `tests/test_<project>_facts.py` — unit tests for the `Fact`/`Facts` classes (formatting, string passthrough, dump/load roundtrip, unknown-name `KeyError`).
- `tests/test_render_docs.py` — smoke tests for `render_docs.py` (`--reference` table, annotate mode, `render_all` writes `.annotated.md` without clobbering `.md`).

### Per-project file inventory the migration must produce
New: `<project>_facts.py`, `render_docs.py`, `docs/<project>_facts.json`, `docs/<project>_guards.json`, `docs/<project>_facts_reference.md`, one `docs/<doc>.md.j2` per reader-facing doc, and the four test files above.
Modified: `<project>_analysis.py` (FACTS calls + dumps + path constants), `generate_report.py` (import + call `render_all()`).

### Gotchas carried over from `home_court`
- The byte-identical render gate does **not** catch under-templating (a bare literal trivially matches itself). Drive completeness with a literal-token grep + `--annotate` review.
- Pick one p-value display convention (`= 0.004` vs `< 0.01`) when templating.
- Derived/reformatted numbers that legitimately don't appear verbatim in `results.md` go in `ALLOW_ABSENT`.
- `results.md` itself is intentionally NOT templated from facts; the render + cross-check already cover the high-stakes cases.
- Leave deliberate editorial roundings literal ("more than 40%", "two-thirds"); `/check-consistency` covers those, not the guard test.

## 4. The list-fact pattern (player_ranking only)

home_court facts are scalars. player_ranking needs ranked lists of player names + z-scores (consensus top-20, wins-predictive top-20, who-each-system-loves). Two options; pick in §5.

- **Option A (recommended): per-item scalar facts.** Register `consensus.top.1.name`, `consensus.top.1.z`, … `consensus.top.5.name`. Templates cite individual ranks: `<< f("consensus.top.1.name") >>`. Most consistent with the existing `Fact`/guard machinery and the cross-check test (each name/number is independently verified against results.md). Verbose but mechanical. Only register the ranks the prose actually cites (likely top 5, not all 20).
- **Option B: a list-valued `Fact`.** Extend `Fact` to accept a list `value` with a join format, render as `"A, B, C, D, E"`. Fewer facts, but breaks the scalar cross-check (the test would need a list-aware branch) and complicates `--annotate`. More code for one project.

Recommendation: Option A, scoped to only the cited ranks. It needs zero changes to the copied infrastructure.

Audit one or more reader-facing docs for factual and numerical consistency. This is the fact-and-figures counterpart to `/voice-review`: voice-review checks prose style, this checks that the numbers are right, the sections agree with each other, and nothing contradicts the source of truth.

Scope: `$ARGUMENTS` is the doc path(s) to check. If empty, check the current project's `docs/<project>_findings.md` and `docs/<project>_summary.md` (the standard doc names from `questions/CLAUDE.md`).

The source of truth is `docs/RESULTS.md` (auto-generated by the analysis pipeline; never edit it by hand). Every figure in the prose must trace back to it.

Steps:

1. Read `docs/RESULTS.md` in full. This is the authoritative output; treat its numbers as correct.

2. Read each target doc in full. Note which `RESULTS.md` section each prose claim draws on.

3. Check, and flag every problem, grouped worst-first:
   - **Wrong or stale numbers**: any figure (record, percentile, margin, SRS, rank, percentage) that does not match `RESULTS.md`. Quote both the doc value and the `RESULTS.md` value.
   - **Internal contradictions**: a claim in one section that conflicts with another section, or with the intro/verdict. Example from a past run: the intro said the Hawks "played below their regular-season rating" while a later section correctly excluded the Hawks for having no independent playoff games.
   - **Claims the data does not support**: a statement stronger than `RESULTS.md` establishes (e.g. "largest in the dataset" when the data only ranks within a subset; "most improved team" when only a few teams were measured). Distinguish "most improved of the opponents" from "most improved in the field".
   - **Same quantity, different numbers**: the same thing cited with two values in different places without explanation (e.g. a playoff rating shown as +14.48 in one section and +15.13 in another because two different definitions are used). Either reconcile to one number or add a one-line note explaining why they differ.
   - **Intro/verdict vs. body**: every headline claim in the up-front section must be developed and matched by the section it summarizes.
   - **Cross-references**: every `§N` pointer resolves to the section it names; section numbers are sequential with no gaps or repeats.

4. Check coverage (summary docs only): does `<project>_summary.md` cover the document's main sections, and does it agree with the intro? Note any major finding omitted; say plainly whether the omission is acceptable for a one-pager (secondary context is fine to drop) or a gap.

5. For each flag, quote the phrase with its line number, name the conflict (vs. which `RESULTS.md` line or which other section), and propose a specific rewrite.

6. Present the review grouped worst-first. Do NOT edit yet. State explicitly what you verified as correct, not just what's wrong.

7. Ask whether to apply. On confirmation, apply the edits, then regenerate the affected outputs following the cascade rules in `questions/CLAUDE.md`: run `/regen` (or `bash /Users/justin/code/nba_analysis/questions/regen_docs.sh`) from the project root. If a number in the prose changed, double-check the change against `RESULTS.md` once more before regenerating.

Note: this command does not change `RESULTS.md`. If the prose is right and `RESULTS.md` looks wrong, the fix is in the analysis code (`<project>_analysis.py`), not the doc, re-run the pipeline.
